# local_mycoursesfilter

`local_mycoursesfilter` is a Moodle local plugin that renders a dedicated, linkable “My courses” page for the current user and applies filter, sort, category, and display options from URL parameters.

It is intended for dashboard shortcuts, course navigation links, deep links from other local plugins, portals, and external integrations where a pre-filtered course overview is useful.

## Current development status

The plugin is already usable for controlled production scenarios, but it is not yet fully hardened as a broadly reusable Moodle ecosystem plugin.

### What is already in good shape

- Dedicated standalone page at `/local/mycoursesfilter/index.php`
- Core-aligned layout based on Moodle's my/courses visual patterns and Mustache templates
- User-persistent toolbar preferences for filter, sort, direction, and display mode
- Safe handling for local return URLs, including `returnurl=this`
- Category shortcuts and recursive category expansion
- PHPUnit and Behat coverage for core request-handling and filtering behaviour
- CI workflow for multiple Moodle, PHP, and database combinations

### What still limits maturity

- The plugin stores site-wide user preferences for toolbar state and therefore needs a Privacy API provider
- It has no own database tables, but still reads personal course-related metadata from core subsystems such as enrolment, favourites, completion, and hidden-course preferences
- It should still be treated as a focused integration plugin rather than a fully generic drop-in replacement for core my/courses

## Security and suitability for production systems

The current implementation is reasonably safe for production use **when deployed in a controlled Moodle environment** and when URLs are generated by trusted Moodle code.

### Positive security properties

- Request parameters are normalised with Moodle `optional_param()` cleaning or strict internal token parsing
- Category selectors are restricted to numeric ids and the keywords `this`, `parent`, and `children`
- Database access uses Moodle DML APIs with bound parameters rather than raw interpolated SQL
- `returnurl` is restricted to local URLs and rejects foreign targets
- `returnurl=this` only resolves when the HTTP referrer belongs to the same Moodle site
- Explicit local return URLs are normalised against the Moodle base path, which makes subdirectory installations safer

### Remaining security and operations recommendations

- Prefer generating plugin links from trusted Moodle renderers, blocks, or local plugins rather than allowing arbitrary user-supplied URLs
- Keep the dedicated capability check in place and review role assignments for that capability during rollout
- Review the exported privacy metadata during deployment on sites with strict compliance requirements
- Keep Behat and PHPUnit in CI mandatory, not advisory, for future changes
- Remove stale duplicate CI files outside `.github/workflows/` and editor metadata from the repository, so only the active workflow remains authoritative

## Feature overview

- Shows only courses in which the current user is enrolled
- Supports free-text search over `fullname` and `shortname`
- Supports category filtering, including multiple ids and contextual category keywords
- Supports tag filtering by tag id or name
- Supports custom field filtering by shortname and value
- Supports derived status filters:
  - `all`
  - `notstarted`
  - `inprogress`
  - `completed`
  - `favourites`
  - `hidden`
- Supports sorting by:
  - last access
  - full name
  - short name
  - latest enrolment
- Supports display modes:
  - `card`
  - `list`
  - `summary`
- Supports persistent toolbar state per user
- Supports optional safe Back button rendering via `returnurl`
- Supports runtime page title override via `title` when enabled in plugin settings

## Installation

1. Copy the plugin into `local/mycoursesfilter`
2. Visit `Site administration > Notifications`
3. Review the plugin settings
4. Add links to `/local/mycoursesfilter/index.php` where needed

## Plugin settings

The plugin currently provides the following site-level settings:

- **Default category scope**
  - `recursive`: selected categories automatically include descendant categories
  - `only`: only the explicitly selected categories are used
- **Default page title**
  - when empty, the plugin falls back to the language string
- **Allow title override from URL**
  - when enabled, the `title` URL parameter may override the configured default title


## Privacy

The plugin does **not** create its own database tables and does not persist course data itself.

However, it **does** store four site-wide user preferences for toolbar persistence in Moodle's user preferences subsystem:

- `local_mycoursesfilter_filter`
- `local_mycoursesfilter_sort`
- `local_mycoursesfilter_dir`
- `local_mycoursesfilter_view`

A Privacy API provider is therefore required and included. The provider declares these preferences and exports them for user data requests.

## URL parameters

Base URL:

```text
/local/mycoursesfilter/index.php
```

### Supported request parameters

| Parameter | Type | Description |
|---|---|---|
| `q` | text | Case-insensitive search against course fullname and shortname |
| `tag` | text/integer | Course tag name or numeric tag id |
| `catid` | text | Category selector; supports ids, comma-separated id lists, and the keywords `this`, `parent`, `children` |
| `field` | text | Course custom field shortname |
| `value` | text | Expected custom field value; when omitted with `field`, any non-empty value matches |
| `filter` | text | One of `all`, `notstarted`, `inprogress`, `completed`, `favourites`, `hidden` |
| `status` | text | Legacy alias for `filter`; `any` maps to `all` |
| `sort` | text | One of `lastaccess`, `alpha`, `shortname`, `lastenrolled` |
| `dir` | text | One of `asc`, `desc` |
| `view` | text | One of `card`, `list`, `summary` |
| `returnurl` | local URL | Optional safe Back target; also accepts `this` |
| `title` | text | Optional page title override when enabled in plugin settings |
| `courseid` | integer | Optional explicit source course id used for contextual category keywords |
| `only` | bool | Forces non-recursive category interpretation |
| `recursive` | bool | Forces recursive category interpretation |

### Category parameter behaviour

`catid` accepts the following forms:

- single id, for example `catid=2`
- multiple ids, for example `catid=2,3,4,5`
- contextual keywords:
  - `this`: the category of the source course
  - `parent`: the parent category of the source course category
  - `children`: the direct child categories below the source course category
- mixed forms, for example `catid=parent,12,children`

### Recursive category handling

By default, category handling follows the plugin setting:

- `recursive` includes the selected categories **and all descendants**
- `only` includes **only** the selected categories

Request switches override the default:

- `only=1` disables recursive expansion
- `recursive=1` forces recursive expansion

If both are present, `only` wins.

### Return URL behaviour

`returnurl` is optional. When present and valid, the page renders a Back button.

Supported forms:

- explicit local path, for example:

```text
returnurl=%2Fmy%2Fcourses.php
```

- contextual referrer mapping:

```text
returnurl=this
```

`returnurl=this` maps to the current page referrer **only if the referrer belongs to the same Moodle site**.

### Title behaviour

The page title is resolved in this order:

1. `title` request parameter, when URL title overrides are enabled
2. configured plugin default title
3. language string fallback

## Example URLs

Filter by course name:

```text
/local/mycoursesfilter/index.php?q=biology
```

Filter by multiple categories:

```text
/local/mycoursesfilter/index.php?catid=2,3,4,5
```

Use contextual category selection from the current course:

```text
/local/mycoursesfilter/index.php?catid=parent&returnurl=this
```

Force non-recursive category matching:

```text
/local/mycoursesfilter/index.php?catid=12&only=1
```

Override the page title:

```text
/local/mycoursesfilter/index.php?filter=inprogress&title=Current%20learning
```

Filter by custom field and show a Back button:

```text
/local/mycoursesfilter/index.php?field=deliverymode&value=online&returnurl=%2Fmy%2Fcourses.php
```

## Access model

The page requires an authenticated user.

Access is additionally protected through the capability `local/mycoursesfilter:view`, which is checked at system context in `index.php`.

This removes the earlier hard-coded dependency on the `student` role and makes the plugin reusable in sites with different enrolment and role models.

## Status filtering logic

The status filter combines Moodle completion metadata and fallback activity signals:

- `completed`: completion enabled and the course is completed
- `notstarted`: not started according to completion data, or no recorded access when completion is unavailable
- `inprogress`: started but not completed, or recorded access without completion when completion is unavailable
- `all`: no status restriction

Because Moodle completion setups differ across sites, these states should be understood as practical UI buckets derived from the best available metadata.

## Rendering approach

The plugin uses dedicated Mustache templates in `local/mycoursesfilter/templates/` and exports a compact card/list/summary context in PHP. The goal is to stay visually close to `my/courses.php` while still allowing link-driven pre-filtering and a responsive non-carousel card layout.
